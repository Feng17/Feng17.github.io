Git基本操作

发表于 2016-11-03

Git与Github

Git是分布式版本控制系统。我们可以在本地新建一个目录，用git命令初始化后这个目录里面的所有文件都可以被Git管理起来，每个文件的修改，添加或删除，Git都能够查看到。

Github 是现在广泛流行的代码托管平台，很多开源项目都把代码放在这里。我们可以使用git命令上传修改代码或者将别人放在GitHub上的代码克隆到本地。

本地 Git 项目与远程的 GitHub 建立联系

我们如果要上传或下载代码，肯定本地Git仓库要与我们在GitHub上的仓库建立联系，二者之间的传输是SSH加密的，所以我们要创建SSH Key。

执行后我们会在用户主目录下可以在用户主目录里找到.ssh目录，里面有id_rsa和id_rsa.pub两个文件，这两个就是SSH Key的秘钥对，id_rsa是私钥，不能泄露出去，id_rsa.pub是公钥，可以放心地告诉任何人。接着在Github设置里将id_rsa.pub文件的内容粘贴进SSH Keys里，这样我们本地就可以与远端GitHub建立联系了。为什么GitHub需要SSH Key因为GitHub需要识别出你的提交确实是你推送的，而不是别人冒充的，而Git支持SSH协议，所以，GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。

GitHub for Windows

我第一次使用git时使用了较为简单的操作，下载了GitHub for Windows 安装后桌面上会有2个图标，一个是GitHub的图形化客户端，另一个是Git Shell。图形化客户端可以代替一些git命令的操作，当第一次登陆GitHub的图形化客户端时我们会输入昵称和邮箱，客户端就会帮我们填写的配置成git的全局配置，并且此时我们填写的邮箱也会收到GitHub发来的邮件表示一个SSH Key添加进我们的账户。Shell是linux/Unix系统的外壳，也可以理解为命令行入口，就是你输入并执行命令行的地方，常用的bash是Shell的一种，Git Shell打开默认是Power Shell,可以在GitHub图形化客户端的Options选项里修改为Git Bash或Cmd,Options里也可以修改我们的Git全局配置。

Git仓库

打开命令行终端，进入项目所在的本地目录，将目录初始化为一个 Git 项目

此时会在目录中创建一个 .git 隐藏文件夹,以 . 开头的文件都是隐藏文件可以使用ls -a 命令查看隐藏文件。本地仓库由 git 维护的三棵“树”组成。第一个是你的 工作目录，它持有实际文件；第二个是 暂存区（Index），它像个缓存区域，临时保存你的改动；最后是 HEAD，它指向你最后一次提交的结果。

文件操作常用命令

ls -a 命令查看隐藏文件 pwd 获取当前路径 cd .. 返回上一级目录 cd ~ 返回’home’目录

mv 源目录文件 目的目录 移动文件 cat test.txt 查看test文本文件 cp test family 复制test文件到family目录中

touch test 创建test空白文件 rm test 删除test文件 mkdir family 新建family目录

添加和提交到仓库

放进新的Git仓库(把它们添加到暂存区) 可以添加多个文件 也可用命令 git add -A 添加所有文件

将代码提交到仓库，此时改动已经提交到了HEAD。若有修改git status命令可以让我们时刻掌握仓库当前的状态，是否被修改过。

git diff 命令可以查看修改内容。

关联到远程仓库

推送到github仓库

本地master分支的最新修改推送至GitHub,也可以把 master 换成你想要推送的任何分支。

以上基本的将本地项目提交到远端的操作就完成了。

分支

你创建仓库的时候，master是默认的分支。在其他分支上进行开发，完成后再将它们合并到主分支上。每次提交，master分支都会向前移动一步，当我们创建新的分支如dev后对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变。

git checkout命令加上-b参数表示创建并切换到dev分支。git branch命令会列出所有分支，当前分支前面会标一个*号。我们可以在dev分支上进行修改并提交到仓库。除非你将分支推送到远端仓库，不然该分支就不为他人所见git push origin <branch>

若此时我们切换到master主分支刚才我们在dev分支上进行的所有操作将都不见了。

合并分支

将dev分支的工作成果合并到master分支上，此时在查看文件会发现我们所修改的内容重新出现了。

合并完成后，就可以放心地删除dev分支了。

git 会尝试去自动合并改动。但可能并非每次都成功，并可能出现冲突（conflicts）。 这时候就需要修改这些文件来手动合并这些冲突。改完之后，你需要执行git add <file name>以将它们标记为合并成功。

在实际开发中应该充分利用分支，master分支应该是非常稳定，在dev分支上进行开发，并不断可以将dev分支合并到master上。
